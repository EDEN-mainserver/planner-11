## SPEC-001: 크몽 메시지 자동 수집

| 항목 | 내용 |
|---|---|
| **연결 기능** | F-001 |
| **우선순위** | critical |
| **상태** | not_started |

5분 주기로 크몽 메시지함을 주기적으로 폴링하여, 읽지 않은 새로운 문의 메시지를 감지하고 발신자, 내용 등 관련 정보를 수집한다.

### 사용자 시나리오

**주체:** 시스템 (자동)
**전제조건:** 크몽 계정 로그인 세션이 유효한 상태이다.

1. Playwright를 사용하여 크몽 메시지함 페이지에 접속한다.
   → 메시지 목록이 정상적으로 로드된다.
2. 읽지 않은 메시지를 나타내는 DOM 요소를 탐색하고 추출한다.
   → 읽지 않은 메시지 요소의 배열이 반환된다.
3. 각 메시지 요소에서 발신자명, 메시지 내용, 수신 시간 등의 데이터를 파싱한다.
   → 구조화된 메시지 객체 배열이 생성된다.
4. 이미 처리된 메시지 ID와 비교하여 중복을 필터링한다.
   → 새로운 문의 메시지만 남는다.

**완료 후:** 수집된 신규 메시지 데이터가 다음 단계(SPEC-002)로 전달된다.

### 수용 기준

- [ ] 5분 이내에 새 메시지를 감지해야 한다.
- [ ] 이미 처리한 메시지를 중복으로 수집하지 않아야 한다.
- [ ] 세션 만료 시 재로그인을 시도하고, 실패 시 담당자에게 알림을 보내야 한다.

### 기술 구현 상세

**구현 방식:** Python의 Playwright 라이브러리를 사용하여 5분 주기로 크몽 메시지함 URL에 접속한다. 로그인 세션 쿠키를 활용하여 인증을 유지하고, 읽지 않은 메시지에 해당하는 CSS 셀렉터를 통해 데이터를 스크래핑한다. 수집된 메시지 ID는 로컬 파일 또는 DB에 저장하여 중복 수집을 방지한다.

**기술 스택:** Python, Playwright, asyncio

**예상 소요:** 1일

**입력 필드:**
| 필드명 | 타입 | 출처 | 필수 |
|---|---|---|---|
| session_cookie | string | auth_manager 모듈 | ✓ |
| processed_message_ids | array | 로컬 DB 또는 파일 | ✓ |

**출력 필드:**
| 필드명 | 타입 | 목적지 |
|---|---|---|
| message_id | string | SPEC-002, SPEC-005 |
| sender_name | string | SPEC-002, SPEC-004, SPEC-005 |
| message_content | string | SPEC-002, SPEC-003, SPEC-004 |
| received_at | date | SPEC-005 |

**CSS/XPath 선택자:**
- 메시지 목록 컨테이너: `TODO: 실제 DOM 확인 필요` (css)
- 읽지 않은 메시지 항목: `TODO: 실제 DOM 확인 필요` (css)

### 에러 케이스

| 상황 | 처리 방법 | 심각도 |
|---|---|---|
| 크몽 로그인 세션 만료 | 자동 재로그인 시도 (최대 2회), 실패 시 담당자에게 알림을 발송한다. | high |
| 크몽 웹사이트의 DOM 구조 변경으로 인한 셀렉터 실패 | 에러 로그를 기록하고 즉시 담당자에게 알림을 발송한다. | critical |

### 테스트 케이스

| 케이스 | 입력 | 기대 결과 |
|---|---|---|
| 새로운 읽지 않은 메시지가 1건 있는 경우 | 유효한 세션 쿠키 | 1건의 메시지 객체를 포함한 배열을 반환한다. |
| 읽지 않은 메시지가 없는 경우 | 유효한 세션 쿠키 | 빈 배열을 반환하고 정상 종료한다. |
| 세션이 만료된 경우 | 만료된 세션 쿠키 | 재로그인을 시도하고 성공 후 메시지 수집을 계속 진행한다. |

### 메모
크몽은 공식 API를 제공하지 않으므로, 웹 스크래핑 방식은 DOM 구조 변경에 취약하다. 셀렉터는 설정 파일로 분리하여 관리하는 것이 유지보수에 용이하다.

## SPEC-002: AI 기반 문의 유형 분류

| 항목 | 내용 |
|---|---|
| **연결 기능** | F-002 |
| **우선순위** | critical |
| **상태** | not_started |

수집된 크몽 메시지 내용을 LLM API를 통해 분석하여 '가격', '작업기간', '수정횟수', '커스텀', '기타' 등 사전에 정의된 유형으로 자동 분류한다.

### 사용자 시나리오

**주체:** 시스템 (AI)
**전제조건:** SPEC-001에서 신규 메시지 데이터가 전달되었다.

1. 메시지 내용을 분류용 LLM 프롬프트 템플릿에 삽입한다.
   → LLM API 호출을 위한 완전한 프롬프트가 생성된다.
2. 구성된 프롬프트를 LLM API (e.g., OpenAI)로 전송한다.
   → API가 호출되고 응답을 대기한다.
3. LLM으로부터 반환된 JSON 형식의 응답에서 문의 유형(inquiry_type)을 파싱한다.
   → 구조화된 분류 결과 데이터를 획득한다.

**완료 후:** 분류된 문의 유형에 따라 후속 처리(자동 응답 또는 담당자 알림)로 분기된다.

### 수용 기준

- [ ] 분류 정확도가 90% 이상이어야 한다.
- [ ] 분류 불가 시 '기타' 유형으로 처리할 수 있어야 한다.
- [ ] API 응답 시간은 평균 5초 이내여야 한다.

### 기술 구현 상세

**구현 방식:** Python의 `openai` SDK를 사용하여 LLM API를 호출한다. 프롬프트에는 분류할 메시지 내용과 함께, 분류 카테고리 목록 및 JSON 출력 형식을 명시하는 지침을 포함하여 안정적인 결과를 유도한다.

**기술 스택:** Python, OpenAI SDK, httpx

**예상 소요:** 4시간

**입력 필드:**
| 필드명 | 타입 | 출처 | 필수 |
|---|---|---|---|
| message_content | string | SPEC-001 Output | ✓ |

**출력 필드:**
| 필드명 | 타입 | 목적지 |
|---|---|---|
| inquiry_type | string | SPEC-003, SPEC-004, SPEC-005 |

**API 엔드포인트:**
- `POST https://api.openai.com/v1/chat/completions` (인증: api_key)
  - 모델: gpt-4o 또는 유사 모델 사용

**선행 기능:** SPEC-001

### 에러 케이스

| 상황 | 처리 방법 | 심각도 |
|---|---|---|
| LLM API 호출 실패 또는 타임아웃 | 2회 재시도 후 실패 시 '기타' 유형으로 분류하고 담당자에게 알림(SPEC-004)을 보낸다. | medium |
| LLM 응답이 유효한 JSON 형식이 아님 | 정규표현식을 이용해 응답에서 JSON을 추출 시도, 실패 시 '기타' 유형으로 분류하고 로그를 기록한다. | medium |

### 테스트 케이스

| 케이스 | 입력 | 기대 결과 |
|---|---|---|
| 가격 관련 문의 메시지 입력 | '로고 디자인 비용이 궁금합니다.' | inquiry_type: '가격' |
| 모호하고 복합적인 문의 메시지 입력 | '첨부한 이미지처럼 제작하고 싶은데, 기간이랑 견적 좀 알려주세요.' | inquiry_type: '커스텀' |

### 메모
프롬프트 엔지니어링이 분류 정확도에 큰 영향을 미친다. Few-shot 예시를 프롬프트에 포함하여 성능을 개선할 수 있다.

## SPEC-003: 맞춤 응답 자동 생성 및 발송

| 항목 | 내용 |
|---|---|
| **연결 기능** | F-003 |
| **우선순위** | high |
| **상태** | not_started |

AI에 의해 분류된 문의 유형 중 자동 응답이 가능한 경우, 해당 유형에 맞는 템플릿을 기반으로 자연스러운 맞춤 응답을 생성하고 크몽 메시지로 발송한다.

### 사용자 시나리오

**주체:** 시스템 (AI + 자동)
**전제조건:** SPEC-002에서 문의 유형이 '가격', '작업기간' 등 자동 응답 대상으로 분류되었다.

1. 분류된 문의 유형에 해당하는 응답 템플릿을 로드한다.
   → 기본 응답 구조가 준비된다.
2. 템플릿과 원본 문의 내용을 LLM에 전달하여 고객 맞춤형 응답 메시지를 생성한다.
   → 자연스럽고 개인화된 응답 텍스트가 생성된다.
3. Playwright를 사용하여 생성된 응답 메시지를 크몽 메시지 입력창에 입력하고 전송한다.
   → 메시지가 성공적으로 발송된다.

**완료 후:** 고객이 자동 응답 메시지를 수신하고, 발송 결과가 로그 기록(SPEC-005)을 위해 전달된다.

### 수용 기준

- [ ] 응답 생성 및 발송까지 1분 이내에 완료되어야 한다.
- [ ] 가격 문의 시, 응답에 최신 가격 정보가 포함되어야 한다.

### 기술 구현 상세

**구현 방식:** 분류된 `inquiry_type`을 키로 사용하여 YAML 형식의 응답 템플릿을 조회한다. 이 템플릿을 기반으로 LLM이 최종 메시지를 생성하며, 생성된 메시지는 Playwright를 통해 크몽 메시지 입력창에 입력 후 전송 버튼 클릭 액션으로 발송된다.

**기술 스택:** Python, Playwright, OpenAI SDK, PyYAML

**예상 소요:** 1일

**입력 필드:**
| 필드명 | 타입 | 출처 | 필수 |
|---|---|---|---|
| inquiry_type | string | SPEC-002 Output | ✓ |
| message_content | string | SPEC-001 Output | ✓ |
| target_message_id | string | SPEC-001 Output | ✓ |

**출력 필드:**
| 필드명 | 타입 | 목적지 |
|---|---|---|
| response_message | string | 크몽 발송, SPEC-005 로그 |
| sent_status | boolean | SPEC-005 로그 |

**CSS/XPath 선택자:**
- 메시지 입력창: `TODO: 실제 DOM 확인 필요` (css)
- 전송 버튼: `TODO: 실제 DOM 확인 필요` (css)

**선행 기능:** SPEC-001, SPEC-002

### 에러 케이스

| 상황 | 처리 방법 | 심각도 |
|---|---|---|
| 메시지 발송 중 DOM 요소(입력창, 버튼)를 찾지 못함 | 페이지를 새로고침하고 1회 재시도, 실패 시 담당자에게 알린다. | high |
| LLM이 부적절하거나 관련 없는 응답을 생성함 | 응답 내용에 대한 기본 검증 실패 시, 재생성을 1회 요청하고 실패하면 담당자 알림으로 전환한다. | medium |

### 테스트 케이스

| 케이스 | 입력 | 기대 결과 |
|---|---|---|
| '가격 문의' 유형에 대한 자동 응답 | inquiry_type: '가격' | 가격 정보가 포함된 응답이 생성되고 성공적으로 발송된다. |
| 크몽 UI 변경으로 전송 버튼 클릭 실패 | 자동 응답 발송 시도 | 재시도 로직이 동작하고, 최종 실패 시 담당자에게 에러 알림이 전송된다. |

### 메모
응답 템플릿은 외부 파일(e.g., templates.yaml)로 관리하여 비개발자도 쉽게 수정할 수 있도록 구성한다.

## SPEC-004: 담당자 알림 (카카오톡 알림톡)

| 항목 | 내용 |
|---|---|
| **연결 기능** | F-004 |
| **우선순위** | high |
| **상태** | not_started |

'커스텀' 또는 '기타'와 같이 자동 응대가 어려운 문의가 접수되면, 담당자가 신속하게 대응할 수 있도록 카카오톡 알림톡으로 문의 내용을 요약하여 발송한다.

### 사용자 시나리오

**주체:** 시스템 (자동)
**전제조건:** SPEC-002에서 문의 유형이 '커스텀' 또는 '기타'로 분류되었다.

1. 알림 메시지 템플릿에 발신자명, 문의 내용 요약, 크몽 대화방 링크를 채워넣는다.
   → 발송할 알림톡 메시지가 완성된다.
2. 카카오톡 알림톡 API를 호출하여 담당자에게 메시지를 발송한다.
   → API 호출이 성공하고 200 OK 응답을 받는다.

**완료 후:** 담당자가 카카오톡으로 알림을 확인하고 직접 응대를 시작한다.

### 수용 기준

- [ ] 분류 후 1분 이내에 알림이 발송되어야 한다.
- [ ] 알림 메시지에는 발신자, 문의 내용 요약, 크몽 메시지 링크가 포함되어야 한다.

### 기술 구현 상세

**구현 방식:** 카카오톡 알림톡 API(또는 솔루션사 API)의 엔드포인트로 HTTP POST 요청을 보낸다. 요청 본문에는 사전에 승인된 템플릿 코드와 함께 고객 문의 내용, 발신자 등의 변수를 포함한다. API 인증을 위해 헤더에 API 키를 추가한다.

**기술 스택:** Python, httpx

**예상 소요:** 4시간

**입력 필드:**
| 필드명 | 타입 | 출처 | 필수 |
|---|---|---|---|
| message_content | string | SPEC-001 Output | ✓ |
| sender_name | string | SPEC-001 Output | ✓ |
| inquiry_type | string | SPEC-002 Output | ✓ |

**출력 필드:**
| 필드명 | 타입 | 목적지 |
|---|---|---|
| notification_sent | boolean | SPEC-005 로그 |

**API 엔드포인트:**
- `POST TODO: 카카오톡 알림톡 API Endpoint` (인증: api_key)

**선행 기능:** SPEC-001, SPEC-002

### 에러 케이스

| 상황 | 처리 방법 | 심각도 |
|---|---|---|
| 카카오톡 알림톡 API 인증 실패 | 로그에 API 키 오류를 기록하고, 이메일 등 2차 알림 수단(Fallback)으로 담당자에게 알림을 보낸다. | high |
| API 서버 응답 지연 또는 실패 | 30초 간격으로 2회 재시도 후, 실패 시 2차 알림 수단으로 전환한다. | high |

### 테스트 케이스

| 케이스 | 입력 | 기대 결과 |
|---|---|---|
| '커스텀' 유형 문의 발생 | inquiry_type: '커스텀' | 담당자의 카카오톡으로 문의 요약 알림이 성공적으로 수신된다. |
| API 키가 잘못된 경우 | 잘못된 API 키로 API 호출 | 인증 실패 에러가 발생하고, 이메일 등 대체 알림이 발송된다. |

### 메모

## SPEC-005: 응대 내역 로그 기록

| 항목 | 내용 |
|---|---|
| **연결 기능** | F-005 |
| **우선순위** | medium |
| **상태** | not_started |

시스템이 처리한 모든 문의에 대해 수신 시간, 발신자, 문의 유형, 처리 방식(자동 응답/담당자 알림) 등의 데이터를 구글 시트에 자동으로 기록하여 이력을 관리한다.

### 사용자 시나리오

**주체:** 시스템 (자동)
**전제조건:** SPEC-003(자동 응답) 또는 SPEC-004(담당자 알림)의 처리가 완료되었다.

1. 처리 과정에서 수집된 모든 데이터를 하나의 로그 레코드로 조합한다.
   → 구글 시트의 한 행에 해당하는 데이터 배열이 생성된다.
2. Google Sheets API를 사용하여 지정된 스프레드시트에 로그 레코드를 추가한다.
   → API 호출이 성공하고 시트에 새로운 행이 추가된다.

**완료 후:** 모든 응대 내역이 구글 시트에 누적되어 영구적으로 보관된다.

### 수용 기준

- [ ] 수신시간, 발신자, 문의유형, 처리방식, 처리시간이 정확히 기록되어야 한다.
- [ ] 처리 완료 후 1분 이내에 로그가 기록되어야 한다.

### 기술 구현 상세

**구현 방식:** Python용 Google API 클라이언트 라이브러리(`google-api-python-client`)를 사용한다. 서비스 계정 인증 방식을 통해 인증을 처리하고, `spreadsheets.values.append` 메소드를 호출하여 시트에 데이터를 추가한다.

**기술 스택:** Python, google-api-python-client, google-auth

**예상 소요:** 4시간

**입력 필드:**
| 필드명 | 타입 | 출처 | 필수 |
|---|---|---|---|
| received_at | date | SPEC-001 Output | ✓ |
| sender_name | string | SPEC-001 Output | ✓ |
| inquiry_type | string | SPEC-002 Output | ✓ |
| action_taken | string | SPEC-003 또는 SPEC-004의 처리 결과 | ✓ |

**출력 필드:**
| 필드명 | 타입 | 목적지 |
|---|---|---|
| log_status | boolean | 내부 처리 확인용 |

**API 엔드포인트:**
- `POST https://sheets.googleapis.com/v4/spreadsheets/{spreadsheetId}/values/{range}:append` (인증: oauth2)

**선행 기능:** SPEC-001, SPEC-002, SPEC-003, SPEC-004

### 에러 케이스

| 상황 | 처리 방법 | 심각도 |
|---|---|---|
| Google API 인증 실패 (서비스 계정 키 오류) | 에러 로그를 기록하고 담당자에게 키 파일 확인을 요청하는 알림을 보낸다. 로그 데이터는 로컬 파일에 임시 저장한다. | high |
| Google Sheets API 할당량(rate limit) 초과 | 60초 대기 후 재시도하는 로직을 구현한다. | medium |
